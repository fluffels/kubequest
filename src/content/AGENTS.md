# AGENTS.md — Kontext-lokale Regeln für `src/content/` (Content-as-Data)

> **Modul-lokale Arbeitsanweisung.** Diese Datei liest du **nur, wenn du in `src/content/` arbeitest** (Quests/NPCs/Objekte/Checks anlegen oder ändern) — sie hält die sehr detaillierten Content-as-Data-Konventionen, die früher verstreut in der Wurzel-[AGENTS.md](../../AGENTS.md) lagen. So lädt ein Agent nur den Kontext seines Moduls (Kontext-Selektor, #483), statt diese Regeln global mitzuschleppen.
>
> **Verwandt:** die **Modul-Interna** (Loader/Schema/Check-DSL/Entity-Registry im Detail) stehen im Tiefendoc [docs/module/content.md](../../docs/module/content.md); der **Bounded Context** dieses Verzeichnisses (Shared Kernel „Content-Loader") ist in [docs/glossar.md](../../docs/glossar.md) benannt (#477). Die harten **Projekt-Regeln** (Git-Workflow, Tests, Board) bleiben in der Wurzel-[AGENTS.md](../../AGENTS.md) — bei Konflikt gilt die Wurzel.

⭐ **Oberste Regel gilt auch hier:** „Ist das okay, wenn KubeQuest ein Spiel in Stardew-Valley-Größe wird?" Content skaliert nur, wenn er **kein Monolith** ist — pro Region/Geber/NPC eine Datei, wie Stardew. Granularität immer mitdenken, nicht nur das Format.

## Wie man Inhalt hinzufügt (Stand #348/#349/#353/#354, auf `main`)

Quests, NPCs, Smalltalk und NPC-Standplätze sind **Daten** (`src/content/data/*`), kein TS mehr. Konkret, damit es auf Stardew-Größe skaliert (kein Monolith – pro Region/NPC eine Datei, wie Stardew):

### Neue Quest

- Als Objekt in die passende Regionen-Datei `src/content/data/quests/<giver>.json` (eine Datei je Geber/NPC) – **und** ihre ID an der richtigen Stelle in `src/content/data/quest-order.json` eintragen (die Reihenfolge ist load-bearing: `GameState.questIdx` ist ein Index in diese Sequenz, sie folgt NICHT der ID). **Quest-IDs sind sprechende, kebab-case Slugs** (z.B. `k8s-service`, `gitops-app-of-apps`), keine `qN`-Nummern mehr (#354) – frei semantisch wählbar, da die Reihenfolge aus `quest-order.json` kommt.
- **`topic` ist Pflicht (#327):** jede Quest braucht ein `topic`-Feld = die ID eines Themas aus `src/content/data/quest-topics.json` (die Themen-Taxonomie + Anzeigereihenfolge fürs Logbuch-Accordion #326). Das Thema wird **nicht** aus dem Geber abgeleitet (ein Geber kann mehrere Themen geben). Gibt es das Thema noch nicht, erst in `quest-topics.json` an die pädagogisch passende Stelle eintragen. Fehlt `topic` ganz, scheitert der Loader hart; ein unbekanntes oder ein totes (von keiner Quest benutztes) Thema meldet `content/validate.ts` + `content.test.ts`.
- **Sinnvoll eingliedern – auch mittendrin, nicht nur ans Ende.** Die ID an die **pädagogisch passende** Stelle in `quest-order.json` setzen (Lernpfad/Schwierigkeit; der Lernreihenfolge-Wächter `content/learnorder.ts` + `content.test.ts` prüfen, dass Befehle/Karten vor ihrer Nutzung eingeführt sind – also dort ggf. mitziehen).
- **Einfügen/Umsortieren braucht KEINE Save-Migration.** Seit #353/#354 ist der Fortschritt per **Quest-ID** gespeichert (`currentQuestId` + `completedQuests`), nicht per Zahl-Index. Bestehende Stände lösen ihre Quest also weiter korrekt auf, der Index verschiebt sich folgenlos. Wer die Einfügestelle schon passiert hat, wird **nicht** zurückgeschickt (die neue Quest bleibt für ihn ungespielt; neue Spieler erleben sie in Reihenfolge). Das ist genau der Stardew-Scope-Sinn von #353/#354.
- **Save-Migration NUR beim Umbenennen oder Entfernen einer Quest-ID** (weil die ID persistiert ist): **Umbenennen** → alten→neuen Slug in `LEGACY_QUEST_ID_MAP` (`game.ts`) eintragen **und** einen Migrations-Test mit echtem Alt-Stand schreiben (Red-Green); bei Save-**Formatänderung** zusätzlich `CURRENT_SAVE_VERSION` (`store.ts`) bumpen, damit Alt-Stände vor dem Überschreiben gesichert werden. **Entfernen** ist unkritisch: die alte ID bleibt harmlos in `completedQuests` stehen, `currentQuestId` fällt in `sanitizeState` über den Index-Fallback weich zurück. **Grundregel: was auf `main` (live) geht, darf bestehende Spielstände NIE brechen.**
- **`accept`** wird als **String-Pattern** geschrieben (Loader kompiliert → RegExp).
- **`check`** (prüft den Sim-Zustand nach einer Aufgabe) ist seit #411 **deklarative Daten**: eine kleine Regel in der JSON, die `src/content/check-dsl.ts` beim Laden zu einem Prädikat kompiliert – **kein Code-Eintrag mehr** für den Normalfall. Beispiele: `"check": { "some": "services", "where": { "name": "kasse" } }` (Service existiert), `{ "some": "deployments", "where": { "name": "leuchtfeuer", "broken": { "truthy": false } } }` (Deployment heil), `{ "count": "nodes", "cmp": ">", "value": 3 }`, `{ "flag": ["tf", "applied"] }`, `{ "all": [ … ] }` / `any` / `not`. Adressierbare Sammlungen stehen als geschlossene Allowlist in `check-dsl.ts` (`CHECK_COLLECTIONS`); eine unbekannte Sammlung scheitert hart beim Laden. **Nur echte Sonderfälle bleiben Code** in `src/content/checks.ts` (`QUEST_CHECKS`) und werden per **String-Key** referenziert – das sind Bedingungen, die kein deklarativer Zustand sind, sondern eine transiente Aktions-Markierung (z.B. „Spieler hat gerade einen Pod gelöscht", `sim.lastDeletedPod`). Im Zweifel zuerst die DSL prüfen; nur was sie wirklich nicht kann, kommt nach `checks.ts`. Details: [docs/module/content.md](../../docs/module/content.md).

### Variable Platzhalter in Beispielbefehlen (#311)

Einen „ändere-mich"-Wert (selbst gewählter Name, Image, Datei …) schreibt man in Content-Texten (Dialoge, Funk-Aufgaben, Hinweise/`why`, Drills, Quiz `q`/`options`/`explain`, Erklärungen) als **`<token>` in spitzen Klammern** – die CLI-Konvention, z.B. `docker pull <image>`, `docker run -d --name <eigener-name> <image>`, `kubectl describe pod <name>` (`token` = ein Wort aus Buchstaben inkl. Umlauten, Ziffern, Bindestrichen; **Bindestrich statt Unterstrich**). `src/markup.ts` › `fmtCmd` macht daraus beim Rendern ein sichtbares, farbig abgesetztes Badge (die spitzen Klammern bleiben sichtbar) – das ist die **eine Quelle** der Konvention, angewandt an der Render-Grenze (radio/dialog/quiz/questlog/album). Echte HTML-Tags (`<code>`, `<b>`, `<i>` …) bleiben unangetastet. **Wörtliche** spitze Klammern, die KEIN „ändere-mich"-Wert sind (Tool-Ausgaben wie `<none>`/`<sensitive>`, Git-Konfliktmarker `<<<<<<<`), gehören weiterhin als `&lt;…&gt;` in die Daten – sie sollen NICHT als Platzhalter erscheinen. Ein Wächter-Test (`content.test.ts`) meldet Platzhalter, die `fmtCmd` nicht sichtbar macht.

### Neuer NPC

Identität (Name/Titel/Sprite) in `data/npcs.json`, Smalltalk in `data/smalltalk.json` – **und** den Standplatz (welche Karte, welche Kachel) in `data/entities.json` › `npcs` (Entity-Registry #349: `{ id, map, x, y }`). Damit ist ein NPC nur noch ein Daten-Eintrag, kein Code-Edit: `world.ts` (Hafen) und die Insel-Szenen loopen über `npcSpawnsForMap(<karte>)`. (Ausnahme: die schiff-relative Quiz-Krabbe Kralle steht bewusst nicht in der Registry, sondern an `SHIP_KRALLE`.)

### Neues platziertes Objekt/Interaktable

(Quest-Trigger, Kran, Container, Schild, Deko-Bauwerk …): Eintrag in `data/entities.json` › `objects` (Entity-Registry #357: `{ id, map, x, y, type, sprite?/label?, w?/h? }`). `type`: `quest_trigger` (begehbar, braucht `label` fürs Schild), `prop` (solides 1×1-Sprite, braucht `sprite`) oder `tower` (solides w×h-Bauwerk mit eigener Darstellung, braucht `sprite`). Geometrie-Module + Insel-Szenen loopen über `objectsForMap(<karte>)` (solide Objekte → Kachel-Solid via `objectFootprint`), Geometrie-Konstanten leiten sich per `objectForId` ab – ein neues Objekt ist nur ein Daten-Eintrag, kein Code-Edit. Das **Render-Tuning** (Sprite-Skalierung/Schatten) liegt bewusst in der Präsentation (`scenes/shared.ts` › `PROP_RENDER`, ein neuer Sprite-Typ = eine Zeile), die **Sim-Mechanik** eines Triggers bleibt Code. Prozedural gestreute Deko (`decor.ts`, Bäume/Felsen/Lager-Güter) ist KEIN Registry-Objekt.

### Der validierende Loader

`src/content/loader.ts` + `src/content/entities.ts` lädt + prüft alles beim Start und wirft bei kaputten Daten explizit (`ContentValidationError`). **Granularität immer mitdenken:** wird eine Regionen-Datei zu groß, in sinnvolle Unterdateien splitten – eine Umstellung ist nur etwas wert, wenn sie dem Wachstum standhält.
