## Architektur

Vite + TypeScript + ES-Module. `index.html` lГӨdt nur `src/main.ts`, Vite bГјndelt den Rest. Phaser 3 als npm-Paket. **Zwei Build-Wege aus derselben Quelle** (Ticket #58, konfiguriert in [`vite.config.ts`](vite.config.ts) Гјber den Vite-`mode`): `npm run build` ist der normale Multi-File-Host-Build nach `dist/` (Assets als eigene, einzeln cachebare Dateien вҖ“ zum Ausliefern Гјber einen Webserver); `npm run build:offline` (`vite build --mode offline`) erzeugt via `vite-plugin-singlefile` eine self-contained `dist-offline/index.html` (alle Assets inline als Data-URI вҖ“ das Doppelklick-Offline-Feature). Das Single-File-Plugin ist bewusst **nur** im Offline-Mode aktiv, sonst wГӨre auch der Host-Build wieder eager-inline.

**Warum Phaser 3 (und nicht Godot/Unity/MonoGame)?** Bewusst festgehalten als ADR: [`docs/adr/0001-engine-phaser.md`](docs/adr/0001-engine-phaser.md). Kurz: Phaser bedient den Kern-Wert (offline, eine Datei, verschenkbar, Lern-Tool); ein Engine-Wechsel wГӨre ein kompletter Rewrite fГјr marginalen Gewinn. Native Distribution kommt Гјber einen Wrapper (#83 Tauri), nicht Гјber einen Engine-Wechsel. Re-Evaluierungs-Trigger stehen im ADR. **Vor erneuter вҖһEngine wechseln?"-Diskussion bitte dorthin verweisen.**

**Warum kein Multiplayer/Co-op?** Bewusst festgehalten als ADR: [`docs/adr/0003-multiplayer-coop-out-of-scope.md`](docs/adr/0003-multiplayer-coop-out-of-scope.md). Kurz: KubeQuest bleibt Single-Player вҖ“ Co-op erzwingt den Backend-Stack (Server/Netcode/DB), den wir bewusst nicht bauen (#85), und bricht den Offline-eine-Datei-Wert; der Lern-Kern ist solo. Architektur nicht dafГјr verbauen, aber auch nichts proaktiv dafГјr bauen. **Vor erneuter вҖһSollten wir Co-op machen?"-Diskussion bitte dorthin verweisen.**

**Langfristige Skalierungsstrategie (Stardew-Scope)** festgehalten als ADR: [`docs/adr/0004-skalierungs-fundament.md`](docs/adr/0004-skalierungs-fundament.md). Das Ziel ist ein wirklich groГҹes Spiel (100+ Quests, 50+ NPCs, viele Welten) вҖ“ dafГјr fehlen drei Fundamente, die jetzt gebaut werden: **Content-as-Data** (#348 вҖ“ Quests/Dialoge als JSON/YAML, nicht TS-Code), **Entity-Registry** (#349 вҖ“ datengeВӯsteuerter NPC/Objekt-Katalog), **IndexedDB** (#350 вҖ“ kein localStorage-Limit mehr). Phaser bleibt die richtige Wahl, solange diese Fundamente vorhanden sind; der neue Re-Eval-Trigger steht in ADR 0001. **Kein neues Content-Ticket anlegen, das Quests/NPCs als TS-Objekt-Literal hard-codiert** вҖ“ immer erst #348 vorziehen oder als Dependency markieren.

> **Content-as-Data вҖ“ wie man jetzt Inhalt hinzufГјgt (Stand #348/#349, auf Branch `umbau/skalierungs-fundament`).** Quests, NPCs, Smalltalk und NPC-StandplГӨtze sind **Daten** (`src/content/data/*`), kein TS mehr. Konkret, damit es auf Stardew-GrГ¶Гҹe skaliert (kein Monolith вҖ“ pro Region/NPC eine Datei, wie Stardew):
> - **Neue Quest:** als Objekt in die passende Regionen-Datei `src/content/data/quests/<giver>.json` (eine Datei je Geber/NPC) вҖ“ **und** ihre ID an der richtigen Stelle in `src/content/data/quest-order.json` eintragen (die Reihenfolge ist load-bearing: `GameState.questIdx` ist ein Index in diese Sequenz, sie folgt NICHT der ID-Nummer).
> - **`accept`** wird als **String-Pattern** geschrieben (Loader kompiliert вҶ’ RegExp); **`check`** (Sim-PrГӨdikat = Mechanik) kommt als benannte Funktion in `src/content/checks.ts` und wird per Key referenziert.
> - **Neuer NPC:** IdentitГӨt (Name/Titel/Sprite) in `data/npcs.json`, Smalltalk in `data/smalltalk.json` вҖ“ **und** den Standplatz (welche Karte, welche Kachel) in `data/entities.json` (Entity-Registry #349: `{ id, map, x, y }`). > - Der validierende Loader (`src/content/loader.ts` + `src/content/entities.ts`) lГӨdt + prГјft alles beim Start und wirft bei kaputten Daten explizit (`ContentValidationError`). **GranularitГӨt immer mitdenken:** wird eine Regionen-Datei zu groГҹ, in sinnvolle Unterdateien splitten вҖ“ eine Umstellung ist nur etwas wert, wenn sie dem Wachstum standhГӨlt.

**Schichtung** вҖ“ pure DomГӨne вҶ” Anwendung вҶ” PrГӨsentation, Persistenz entkoppelt. - **pure DomГӨne** (kein Phaser, voll unit-testbar): `sim.ts` (Cluster-Simulator), `content.ts` (Fassade Гјber `src/content/*`: Quests/Drills/Quiz/NPCs/Progression/Minispiel), `world.ts`, `decor.ts`, `clock.ts`, `pixelfont.ts` (Glyphen-Daten der In-Welt-Bitmap-Font, #188).

## Wo die TODOs leben

Der Backlog wird als **GitHub Issues** gefГјhrt, gruppiert in einem **GitHub Project-Board** (Kanban). PrioritГӨten als Labels: `prio:hoch` рҹ”ҙ / `prio:mittel` рҹҹ  / `prio:niedrig` рҹҹЎ. **Die `prio:`-Labels steuern die Auswahlreihenfolge:** erst alle `prio:hoch`, dann `prio:mittel`, dann `prio:niedrig` (Issues **ohne** prio-Label kommen ganz zuletzt) вҖ“ **innerhalb derselben Stufe** entscheidet die niedrigste Nummer.

**Auswahl des nГӨchsten Tickets вҖ“ schnell und deterministisch, NICHT lange suchen, NICHT abwГӨgen.** Hol dir zuerst die **vollstГӨndige** offene Liste (`gh issue list --state open --limit 500 вҖҰ` вҖ“ **ohne `--limit` nur die 30 neuesten**, siehe Kollisionsschutz unten), dann sortiere die **freien** Issues (nicht assigned, kein offener PR/Branch/Worktree, kein вҖһin progress") nach **PrioritГӨt zuerst** (`prio:hoch` вҶ’ `prio:mittel` вҶ’ `prio:niedrig` вҶ’ ohne prio-Label) und **innerhalb derselben PrioritГӨt nach NIEDRIGSTER Nummer** вҖ“ nimm das oberste. Ist es schon geclaimt, nimm das nГӨchste nach derselben Sortierung. Das bleibt **deterministisch und ohne inhaltliches AbwГӨgen**: nur prio-Label + Nummer entscheiden, **nicht der Inhalt** des Tickets. Nach jedem Ticket `npm test` + `npm run typecheck` grГјn halten + im Browser verifizieren, dann das Issue schlieГҹen.

**Kollisionsschutz bei parallelen Agenten (mehrere Chats gleichzeitig).** Jedes Ticket wird auf einem eigenen Branch + Worktree bearbeitet (`git worktree add -b <typ>/<nr>-<slug> ../kubequest-wt-<nr> main`). Damit sich zwei Agenten nicht dasselbe Ticket greifen:
- **Immer die GANZE offene Liste holen, sonst greifst du das falsche Ticket.** вҡ пёҸ `gh issue list` liefert **ohne `--limit` nur die 30 _neuesten_ Issues** (hГ¶chste Nummern, `created`-desc). Genau die niedrig nummerierten Tickets, die du nach der Sortierung zuerst nehmen sollst, fallen dann raus und sind unsichtbar. Darum **vor der Auswahl immer `--state open --limit 500`** mitgeben: `gh issue list --state open --limit 500 --json number,title,assignees,labels` (zusГӨtzlich `git worktree list` + `git branch -a` als Gegencheck). Die `labels` brauchst du fГјr die Prio-Sortierung (siehe oben).
- **Fertig sortierte Auswahlliste in einem Befehl** вҖ“ holt alle offenen, **freien** Issues (kein Assignee), gruppiert nach Prio (`hoch`вҶ’`mittel`вҶ’`niedrig`вҶ’ohne) und sortiert innerhalb der Gruppe nach niedrigster Nummer. **Die oberste Zeile ist вҖһdran":**
  ```bash
  gh issue list --state open --limit 500 --json number,title,assignees,labels --jq '
    map(select((.assignees|length)==0))
    | map(.prio = (if any(.labels[];.name=="prio:hoch") then 0 elif any(.labels[];.name=="prio:mittel") then 1 elif any(.labels[];.name=="prio:niedrig") then 2 else 3 end))
    | sort_by(.prio, .number)
    | .[] | "\(["hoch","mittel","niedrig","ohne"][.prio])\t#\(.number)\t\(.title)"'
  ```
  (Branch/Worktree-Gegencheck trotzdem machen: ein Branch/Worktree ohne Assignee ist selten, aber `git worktree list` + `git branch -a` schlieГҹt das aus.)
- **Beim Start sofort self-assignen:** `gh issue edit <nr> --add-assignee @me`. Das ist der вҖһin Arbeit"-Marker, den jeder andere Chat sieht вҖ” der einzige Zustand, den ein paralleler Agent sonst nicht erkennen kann (ein nur вҖһim Kopf" gewГӨhltes Ticket ohne Branch ist unsichtbar).
- Am Ende: nach `main` mergen, Worktree entfernen (`git worktree remove`), Branch lГ¶schen, Issue schlieГҹen (schlieГҹt den Assignee implizit mit ab).
- **`node_modules` im Worktree:** Im frischen Worktree fehlt `node_modules`. Lass dort einfach einmal `npm install` laufen (ist gitignored, eigener Stand pro Worktree). вҡ пёҸ **Verlinke NICHT** das `node_modules` des Hauptrepos per Junction/Symlink (`mklink /J`, `ln -s`): `git worktree remove --force` folgt dem Link rekursiv und **leert dann die echten `node_modules` des Hauptrepos** (musste schon mit `npm install` repariert werden). Wer trotzdem verlinkt, **muss die Junction zwingend VOR** `git worktree remove` lГ¶sen (`cmd /c rmdir <pfad>\node_modules` bzw. `rm node_modules` вҖ“ das lГ¶scht nur den Link, nicht das Ziel).

**Der Agent managt das Board (nur kubequest).** Die Maintainerin hat die Issue-Verwaltung hier an den Agenten delegiert. Das heiГҹt konkret:
- **GitHub ist die Single Source of Truth fГјr den Stand.** Was erledigt ist, wird sofort dort geschlossen (mit kurzem Ergebnis-Kommentar) вҖ“ nicht nur im Chat berichten. Den Board-Status aktuell halten.
- **Selbst Issues verwalten** ohne RГјckfrage: `gh issue close`, kommentieren, Labels setzen, **neue Tickets schreiben, wenn etwas auffГӨllt** (Bug, LГјcke, Tech-Debt, Idee). Lieber ein Ticket zu viel als verlorenes Wissen.
- **Zu groГҹes Ticket (Epic/Phase) вҶ’ aufteilen statt umsetzen.** Ist ein geclaimtes Issue ein Epic / eine ganze Phase / вҖһFar-Future" und nicht in EINER Session vollstГӨndig umsetz- und schlieГҹbar (z.B. вҖһPhase 4: GitOps-Archipel"): **nicht selbst implementieren.** Stattdessen in **viele konkrete, session-groГҹe Kindertickets** zerlegen und anlegen (jeweils **ohne Assignee** вҖ” der Assignee ist der вҖһin Arbeit"-Marker, neue Tickets sind frei; passende `area:`/`prio:`-Labels setzen), im Epic einen **Гңbersichts-Kommentar** mit den Kindernummern + sinnvoller Reihenfolge posten und das Epic dann **auf done schlieГҹen** (`gh issue close <nr> --reason completed`, **nicht lГ¶schen** вҖ” der Datensatz bleibt als erledigtes Aufteilungs-Ticket). Kein Worktree/Branch/Merge nГ¶tig (kein Code).
- **Auswahlreihenfolge ist fix, nicht Ermessen.** Das nГӨchste Ticket ergibt sich immer aus der festen Sortierung *PrioritГӨt (`hoch` вҶ’ `mittel` вҶ’ `niedrig` вҶ’ ohne) und dann niedrigste Nummer* (siehe oben) вҖ“ der Agent wГӨgt nicht ab und sucht nicht nach Inhalt aus. Damit ist eindeutig bestimmt, welches Ticket вҖһdran" ist; der Agent labelt und pflegt das Board weiterhin (neue Tickets schreiben, labeln, schlieГҹen) вҖ“ und das prio-Label ist jetzt genau der Hebel, der die Reihenfolge steuert.
- **Git ist hier voll freigegeben** (anders als sonst): committen вҶ’ nach `main` mergen вҶ’ **`git push origin main`** вҶ’ Branch/Worktree aufrГӨumen вҶ’ Issue schlieГҹen darf der Agent in kubequest komplett selbst fahren вҖ” **Push inklusive, ohne Extra-Nachfrage**. вҡ пёҸ Das gilt **nur fГјr kubequest**; in **allen anderen Projekten** bleibt committen/pushen/deployen strikt tabu. Issues erst schlieГҹen, wenn die Arbeit fertig + getestet (`npm test` grГјn) + im Browser verifiziert ist; falls ihr Review spГӨter was findet, einfach wieder aufmachen.

Issues ansehen: `gh issue list --state open --limit 500` (ohne `--limit` nur die 30 neuesten). Board: `gh project list --owner fluffels`.

## Forum-Eingang (Discussions) bearbeiten

Das **Forum** sind die **GitHub Discussions** des Repos. Damit eine Forum-Nachricht im Board nicht untergeht, ist das in **zwei HГӨlften** aufgeteilt:

1. **Automatisch (flaggen) вҖ“ GitHub Action [`.github/workflows/forum-inbox.yml`](.github/workflows/forum-inbox.yml).** Trigger: `discussion`/`discussion_comment` (created). FГјr **jede** neue Forum-Nachricht (neuer Thread *oder* neue Antwort) wird genau **ein** offenes `prio:hoch`+`forum`-Issue вҖһForum #N: вҖҰ" angelegt (N = Discussion-Nummer, mit Thread-Link); weitere AktivitГӨt im selben Thread hГӨngt nur einen Kommentar an das bestehende Inbox-Issue, statt zu duplizieren. Posts des Repo-Owners (`fluffels`) werden ausgefiltert, sonst lГ¶st jede selbst gepostete Antwort eine neue Inbox aus. Die Action **antwortet nicht** und legt **kein** echtes Bug-/Feature-Ticket an вҖ“ das ist bewusst der interaktive Teil. вҡ пёҸ `discussion`-Trigger feuern nur, wenn die Workflow-Datei auf `main` liegt.
2. **Interaktiv (antworten + triagieren) вҖ“ mit Freigabe der Maintainerin, per `/forum`-Skill.** Den Ablauf fГјhrt der [`forum`-Skill](.claude/skills/forum/SKILL.md): offene `forum`-Issues holen вҶ’ Discussion-Thread per GraphQL lesen вҶ’ triagieren (Bug / Feature / nur Frage / Spam) вҶ’ **Antworttext entwerfen und der Maintainerin zur Freigabe der Formulierung vorlegen (verbindlicher Stopp вҖ“ nie ungefragt posten)** вҶ’ erst nach OK als `fluffels` im Thread posten (`addDiscussionComment`) вҶ’ bei Bug/Feature ein passend gelabeltes Ticket **ohne Assignee** anlegen, sonst nur als beantwortet schlieГҹen вҶ’ Inbox-Issue mit Ergebnis-Kommentar schlieГҹen und verifizieren.

**Nicht jede Forum-Nachricht wird ein Ticket** вҖ“ das Inbox-Issue ist nur der Flag вҖһbitte ansehen"; ob/welches echte Ticket entsteht, entscheidet die Triage im Skill. Es gelten dieselben AnonymitГӨts-/IdentitГӨts-Regeln wie sonst (posten als `fluffels`, kein Klarname); in der externen Forum-Antwort keine Em-Dashes als Satzverbinder.
